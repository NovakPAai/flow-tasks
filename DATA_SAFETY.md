# DATA_SAFETY.md

> Обязательно к прочтению перед работой с базой данных FlowTask.
> AI-агенты (Claude, Cursor) должны следовать этому документу без исключений.

## Деструктивные команды — требуют явного подтверждения

Следующие команды **уничтожают данные необратимо**. Никогда не запускать без явного подтверждения от владельца проекта:

```bash
# ОПАСНО — удаляет ВСЕ данные и накатывает миграции заново
npx prisma migrate reset

# ОПАСНО — удаляет таблицу с данными
DROP TABLE ...;

# ОПАСНО — очищает таблицу без возможности восстановления
TRUNCATE TABLE ...;

# ОПАСНО — удаляет записи (без WHERE удалит всё)
DELETE FROM ...;
```

## Процедура безопасного сброса БД

Если нужен `migrate reset`, сначала сделай бэкап:

```bash
# 1. Бэкап
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql

# 2. Только после бэкапа — сброс
npx prisma migrate reset --force

# 3. Сид
npm run db:seed
```

## Правила для AI-агентов

- **Никогда не запускай деструктивные SQL без явного `yes, destroy data` от пользователя**
- **Никогда не коммить** `.env`, `*.sql` с реальными данными, бэкапы
- При написании миграций — только **аддитивные изменения** (add column, add table)
- Переименование/удаление колонок — только через **safe migration** (add → backfill → drop)
- Перед любым `DROP` — убедиться что колонка/таблица не используется в коде

## Safe migration pattern

```sql
-- НИКОГДА так:
ALTER TABLE users RENAME COLUMN old_name TO new_name;

-- ВСЕГДА так (3 отдельных PR/миграции):
-- 1. Add new column
ALTER TABLE users ADD COLUMN new_name TEXT;
-- 2. Backfill + deploy code that writes both
UPDATE users SET new_name = old_name;
-- 3. Drop old column after full rollout
ALTER TABLE users DROP COLUMN old_name;
```

## Чеклист целостности данных

После любой миграции:

```bash
# Проверить что миграции применились
npx prisma migrate status

# Проверить что схема синхронизирована
npx prisma db push --accept-data-loss  # ТОЛЬКО в dev, никогда в prod

# Проверить что seed проходит
npm run db:seed
```

## Чувствительные данные

FlowTask хранит:
- Хэши паролей (bcrypt, 12 rounds) — **никогда не логировать raw passwords**
- Хэши refresh-токенов (SHA-256) — **никогда не логировать raw tokens**
- Хэши API-ключей (SHA-256) — raw ключи нигде не хранятся, показываются один раз
- Email-адреса пользователей

При логировании — использовать `logger` из `src/shared/utils/logger.ts`, который автоматически redact'ит чувствительные поля.
