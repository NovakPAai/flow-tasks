Авторизация — логин и пароль
Стандартный способ входа: email + пароль. Доступен всегда, для всех пользователей системы.

Вход в систему (POST /api/auth/login)
1

Защита от брутфорса

Счётчик ошибочных попыток хранится в Redis по ключу email. 5+ неудачных попыток → блокировка на 15 минут. Redis недоступен → счётчик в памяти процесса (graceful degradation).

2

Проверка SSO_ONLY

Если у пользователя в БД ssoOnly=true и он не суперадмин — локальный вход заблокирован (403). Все текущие пользователи имеют ssoOnly=false по умолчанию.

3

Верификация пароля

bcryptjs с 12 раундами соли. Сравнение константного времени — защита от timing-атак.

4

Выдача токенов

Access token (1h) — в теле ответа. Refresh token — в HttpOnly cookie SameSite=Strict. В БД хранится SHA-256 хеш refresh token, а не сам токен.

5

LRU-ротация сессий

Максимум 5 активных сессий на пользователя. При создании 6-й — старейшая удаляется. Операция атомарная (Prisma transaction).

Регистрация
ℹ️
Самостоятельная регистрация создаёт заявку со статусом PENDING. Суперадмин одобряет или отклоняет в панели администрирования. До одобрения войти нельзя. Email автоматически формируется из имени и фамилии (домен задаётся в REGISTRATION_DOMAIN).
Обновление токена (POST /api/auth/refresh)

Axios-интерцептор перехватывает 401, автоматически вызывает /auth/refresh (cookie летит автоматически), получает новый access token и повторяет оригинальный запрос. Пользователь ничего не замечает.

Доступ
SSO — Keycloak и Avanpost новое
Единый вход через корпоративный провайдер идентификации. Поддерживаются Keycloak и Avanpost по стандартному протоколу OIDC/PKCE.
✅
Обратная совместимость: SSO выключено по умолчанию (SSO_ENABLED=false). Все существующие пользователи продолжают заходить по логину и паролю без каких-либо изменений. SSO включается только явным выставлением переменных окружения.
Как работает PKCE-поток
1

Пользователь нажимает «Войти через Keycloak»

Браузер вызывает GET /api/auth/sso/login. Сервер генерирует случайный state (128-битный), PKCE verifier и challenge. Пара сохраняется в Redis на 5 минут. Пользователь перенаправляется на страницу Keycloak.

2

Пользователь вводит корпоративный пароль в Keycloak

Это происходит на странице Keycloak, а не FlowTask. FlowTask никогда не видит корпоративный пароль.

3

Keycloak перенаправляет на /api/auth/sso/callback

Сервер проверяет state через атомарный GETDEL из Redis (одноразовое использование, нет риска повторного использования). Обменивает code на токены Keycloak.

4

JIT Provisioning — автосоздание пользователя

Ищет пользователя по ssoSubjectId = "keycloak:<sub>". Если не найден — ищет по email (нужен email_verified=true). Если не найден — создаёт нового. Гонка параллельных запросов обрабатывается через P2002 + retry.

5

Выдача сессии

Сервер устанавливает refresh token cookie (HttpOnly). Перенаправляет на /?sso_return=1. Фронтенд вызывает /auth/refresh и получает access token — точно так же как после обычного логина. Никаких токенов в URL.

JIT Provisioning — автоматическое создание пользователей
СитуацияЧто происходит
Первый вход через SSO, пользователь новыйСоздаётся новый аккаунт с данными из claims (имя, email). Пароль не задаётся.
Пользователь уже есть в FlowTask (совпадение по email)Аккаунты связываются — добавляется ssoSubjectId. Требуется email_verified=true в токене IdP.
Повторный вход через SSOПоиск по ssoSubjectId — мгновенно, без создания нового аккаунта.
Настройка SSO (переменные окружения)
# Включение SSO SSO_ENABLED=true # false по умолчанию — SSO выключено SSO_ONLY=false # true = запретить локальный вход (кроме суперадминов) # Провайдер OIDC_PROVIDER=keycloak # или avanpost OIDC_ISSUER_URL=https://auth.example.com/realms/flowtask OIDC_CLIENT_ID=flowtask OIDC_CLIENT_SECRET=... OIDC_REDIRECT_URI=https://flowtask.example.com/api/auth/sso/callback OIDC_SCOPE=openid profile email # по умолчанию
⚠️
При SSO_ENABLED=true сервер проверяет на старте, что все OIDC-переменные заданы. Если чего-то нет — process.exit(1) с явным сообщением об ошибке. Неправильная конфигурация не пройдёт незамеченной.
Кнопка SSO на странице входа

Фронтенд запрашивает GET /api/auth/sso/status при загрузке страницы входа. Если enabled: false — кнопка не отображается вообще. Если статус-эндпоинт не отвечает — кнопки тоже нет (fail-safe).

SSO_ENABLED=false

Кнопка «Войти через SSO» не показывается. Только форма email + пароль.

SSO_ENABLED=true

Появляется кнопка «Войти через Keycloak» (или Avanpost). Локальный вход остаётся доступным.

Доступ
2FA / TOTP — второй фактор новое
Двухфакторная аутентификация через TOTP делегируется IdP (Avanpost / Keycloak). FlowTask не хранит TOTP-секреты и не показывает экран ввода кода — это полностью на стороне провайдера идентификации. FlowTask только проверяет результат в токене.
ℹ️
Для кого этот раздел: сотрудники ИБ, настраивающие Keycloak или Avanpost для интеграции с FlowTask. Здесь описано, что именно FlowTask ожидает от IdP и как включить принудительный второй фактор на уровне воркспейса.
Три режима авторизации
#РежимКто выполняет входВторой факторНастройка
1 Локальный FlowTask (email + пароль) нет Только SSO_ENABLED=false
2 SSO Avanpost / Keycloak нет SSO_ENABLED=true, TOTP не настроен в IdP
3 SSO + TOTP Avanpost / Keycloak → экран TOTP IdP ✓ TOTP SSO_ENABLED=true + TOTP настроен в IdP + requireMfa в воркспейсе
⚠️
FlowTask не реализует TOTP самостоятельно для режима SSO. Экран ввода кода, привязка устройства, QR-код — всё это происходит в Avanpost / Keycloak. FlowTask только читает amr claim из OIDC id_token.
Что такое amr claim и зачем он нужен

amr (Authentication Method References) — стандартное поле OIDC id_token (RFC 8176). Содержит список методов аутентификации, которые использовал пользователь при входе. FlowTask читает это поле и сохраняет его в Redis-сессию при SSO-входе.

Значение amrСмыслУдовлетворяет MFA?
pwdТолько парольнет
totpTOTP (Time-based One-Time Password)✓ да
otpOTP (одноразовый код)✓ да
mfaМногофакторная аутентификация (общий)✓ да
hwkАппаратный ключ (YubiKey и др.)✓ да
swkПрограммный ключ (passkey)✓ да
pwd + totpПароль и TOTP (типичный Keycloak)✓ да

Если amr отсутствует в токене или не содержит ни одного из значений выше — FlowTask считает, что второй фактор не пройден.

Настройка Keycloak — включение обязательного TOTP
1

Создать Client для FlowTask

Realm → Clients → Create. Client ID = значение OIDC_CLIENT_ID в FlowTask. Client Protocol = openid-connect. Access Type = confidential. Valid Redirect URIs = значение OIDC_REDIRECT_URI.

2

Включить PKCE

Вкладка Advanced Settings → Proof Key for Code Exchange Code Challenge Method = S256. FlowTask использует PKCE для защиты authorization code flow.

3

Настроить обязательный TOTP (Authentication Flow)

Authentication → Flows → Browser Flow → Duplicate. В копии добавить execution OTP Form с требованием REQUIRED. Привязать новый flow к клиенту: Client → Authentication Flow Overrides → Browser Flow.

4

Убедиться, что amr включён в id_token

Client Scopes → openid → Mappers → Add Mapper → User Attribute. Или проверить, что встроенный mapper amr активен. Keycloak ≥ 18.x включает amr автоматически при использовании OTP.

5

Выдать scope профиля и email

OIDC_SCOPE=openid profile email — FlowTask запрашивает эти scopes. Убедитесь, что клиент имеет доступ к email, email_verified, name / given_name + family_name.

Настройка Avanpost — включение обязательного TOTP
1

Создать OIDC-приложение

Avanpost Admin Console → Applications → New → OIDC. Client ID и Client Secret → в переменные OIDC_CLIENT_ID / OIDC_CLIENT_SECRET. Redirect URI = значение OIDC_REDIRECT_URI.

2

Включить PKCE

В настройках приложения → Token Endpoint Auth Method = none (или client_secret_basic) + включить Code Challenge = S256.

3

Настроить политику MFA

Security Policies → MFA Policy → Add Rule: применить для группы пользователей FlowTask, метод второго фактора = TOTP. Назначить политику на приложение FlowTask.

4

Проверить amr в id_token

Avanpost ≥ 4.x включает amr в id_token автоматически при прохождении TOTP. Для более старых версий — уточнить у вендора или добавить кастомный claim mapper. Ожидаемое значение: ["totp"] или ["password","totp"].

5

Выдать claims профиля

Убедиться, что приложение получает: sub, email, email_verified, name (или given_name + family_name). FlowTask использует эти поля для JIT provisioning.

Переменные окружения для режима SSO + TOTP
# Базовый SSO (обязательно) SSO_ENABLED=true OIDC_PROVIDER=avanpost # или keycloak OIDC_ISSUER_URL=https://sso.company.ru/realms/flowtask OIDC_CLIENT_ID=flowtask-prod OIDC_CLIENT_SECRET=<секрет из IdP> OIDC_REDIRECT_URI=https://flowtask.company.ru/api/auth/sso/callback OIDC_SCOPE=openid profile email # по умолчанию, amr приходит автоматически # Режим SSO-only (опционально) SSO_ONLY=true # запретить локальный вход для всех, кроме суперадминов
⚠️
Дополнительных переменных для TOTP нет — второй фактор настраивается целиком в IdP. FlowTask автоматически читает amr из каждого id_token при SSO-входе.
Принудительный MFA на уровне воркспейса

Владелец воркспейса (OWNER) может включить требование второго фактора для своего воркспейса независимо от других воркспейсов. Это дополнительный уровень защиты поверх настроек IdP.

1

OWNER включает requireMfa в настройках воркспейса

Настройки воркспейса → вкладка Безопасность → переключатель «Обязательная 2FA». После сохранения FlowTask проставляет grace period всем участникам без активного TOTP в сессии.

2

Grace period — 7 дней (настраивается)

Участники, у которых нет amr: totp в текущей сессии, получают 7 дней на настройку TOTP в Avanpost / Keycloak. В это время они видят баннер с обратным отсчётом.

3

После истечения grace period — доступ заблокирован

Попытка войти в воркспейс без amr: totp → 403. Пользователь получает инструкцию настроить TOTP в корпоративном IdP и зайти повторно.

✅
Воркспейсы независимы: включение requireMfa в одном воркспейсе не влияет на другие. Пользователь может работать в воркспейсах без MFA, даже если в одном из них MFA обязателен (при условии, что он не заходит туда без TOTP).
Что происходит при входе — схема проверки
1

Пользователь проходит SSO

FlowTask получает id_token от IdP. Из него извлекается amr (например, ["totp"]) и сохраняется в Redis вместе с сессией.

2

Пользователь открывает воркспейс

Middleware workspaceMfaGuard проверяет: включён ли requireMfa для этого воркспейса?

3

requireMfa = false → пропустить

Проверка amr не выполняется. Запрос продолжается штатно.

4

requireMfa = true → проверить amr

Если в сессии есть amr: totp/otp/mfa/hwk/swk → пропустить. Иначе — проверить grace period. Если grace period активен → пропустить с баннером. Если grace period истёк → 403.

Обновления v1.6 — Gap-13
🆕
AMR enforcement при SSO-входе: FlowTask теперь явно проверяет наличие totp или mfa в amr[] id_token при каждом SSO-входе — не только на уровне воркспейса.

Grace period: настраивается через переменную MFA_GRACE_DAYS (по умолчанию 7). Пользователи без активного TOTP видят баннер с обратным отсчётом.

Confirm-модал при отключении MFA: пользователь обязан подтвердить действие перед отключением второго фактора.

Audit-события: mfa.setup.complete при настройке TOTP, mfa.disable при отключении — оба попадают в AuditLog с SIEM-тегами.

Доступ
API-ключи
Для машинной интеграции без сессий пользователя — вместо JWT используются API-ключи.
1

Создание ключа

Профиль → вкладка «API Keys» → «Создать ключ». Задайте метку (label) и опциональную дату истечения. Ключ формата ft_<64 символа> отображается один раз.

2

Что хранится в БД

Только keyHash (SHA-256), keyPrefix (первые 12 символов для идентификации), label, isActive, expiresAt, lastUsedAt.

3

Использование

Заголовок Authorization: Bearer ft_ваш_ключ. Права = правам создавшего пользователя.

4

Отзыв

Профиль → API Keys → кнопка удаления. Ключ становится недействительным мгновенно.

API — массовые операции
# Массовое обновление (до 100 задач) PATCH /api/boards/:boardId/tasks/bulk { "ids": ["uuid1", "uuid2", ...], // 1–100 UUID задач с этой доски "patch": { "statusId": "uuid", // опционально — должен быть в воркфлоу доски "priority": "HIGH", // HIGH | MEDIUM | LOW | null (очистить) "assigneeId": "userId" // UUID участника или null (очистить) } } # Ответ: { "updated": N } # Массовое удаление (до 100 задач) POST /api/boards/:boardId/tasks/bulk-delete { "ids": ["uuid1", "uuid2", ...] // 1–100 UUID задач с этой доски } # Ответ: { "deleted": N }
⚠️
Задачи из других досок в списке ids молча игнорируются (вернётся updated: 0). Это намеренное поведение — не ошибка. Лимит 100 задач за раз.
Аудит — что записывается в историю
📋
При массовом обновлении каждое изменённое поле записывается в TaskHistory (поле, старое значение, новое значение). При смене статуса также обновляется TaskStatusHistory — закрывается предыдущий интервал и открывается новый. История ведётся точно так же, как для одиночного изменения.
Основной функционал
Глобальный поиск Cmd+K новое
Быстрый поиск задач из любого места системы по нажатию Cmd+K.
Как пользоваться
1

Открыть палитру

Cmd+K (Mac) или Ctrl+K (Windows/Linux). Работает в любом месте системы, кроме открытых модальных окон.

2

Ввести запрос

Начните вводить заголовок задачи или issueKey (например FT-42). Поиск запускается автоматически через 300ms после остановки ввода.

3

Перейти к задаче

Кликните на результат — откроется TaskDrawer с задачей. Поиск закрывается автоматически.

4

Закрыть

Escape или клик вне окна поиска.

API поиска
GET /api/search?q=текст_запроса&limit=20 # Ответ: { "items": [ { "id": "uuid", "issueKey": "FT-42", "title": "Заголовок задачи", "priority": "HIGH", "status": { "name": "В работе", "category": "IN_PROGRESS" }, "board": { "id": "...", "name": "...", "prefix": "FT" }, "workspace": { "id": "...", "name": "...", "slug": "..." } } ] }
💡
Поиск ведётся только по задачам воркспейсов, в которых состоит пользователь. Rate limit: 30 запросов в минуту. Параметр limit — от 1 до 50 (по умолчанию 20).

Обновления v1.6 — Gap-14, 15, 16
🚦

Rate-limit по email (Gap-14)

Лимит аутентификации ключуется по email-адресу, а не по IP. Ротация X-Forwarded-For больше не обходит ограничение. Fallback-бакет no-email для запросов без тела.

🕵️

Email enumeration prevention (Gap-15)

POST /api/auth/register возвращает одинаковый ответ для всех сценариев: существующий email, pending заявка, новый пользователь. hashPassword выполняется всегда — защита от timing side-channel.

🔴

Account disabled check (Gap-16)

JWT middleware проверяет поле isActive при каждом запросе. Отключённый аккаунт немедленно получает 403 ACCOUNT_DISABLED без дальнейшей обработки запроса.

Email и сброс пароля
SMTP-транспорт через nodemailer. Graceful degradation если SMTP не настроен.
Процесс сброса пароля
1

POST /api/auth/forgot-password

Пользователь вводит email. Ответ всегда одинаковый — нельзя угадать есть ли такой пользователь (защита от перечисления).

2

Сервер генерирует token

Случайный 40-байтный токен. В БД сохраняется только его SHA-256 хеш + expiresAt. Оригинал уходит в письме.

3

Письмо с ссылкой

Nodemailer отправляет через SMTP. Если SMTP не настроен — логируется предупреждение, ошибка не пробрасывается пользователю.

4

POST /api/auth/reset-password

Пользователь переходит по ссылке, вводит новый пароль. Сервер ищет SHA-256(token) в БД, проверяет expiresAt и что ещё не использован, меняет пароль.

Переменные SMTP
SMTP_HOST=smtp.example.com SMTP_PORT=587 SMTP_USER=flowtask@example.com SMTP_PASS=... SMTP_FROM="FlowTask <flowtask@example.com>" APP_URL=https://flowtask.example.com # для формирования ссылки в письме
💡
Если SMTP не настроен — функция сброса пароля недоступна. Суперадмин может создать нового пользователя вместо сброса пароля.
